# Source-compatible flavour-pairing data package

A reproducible data foundation for flavour-pairing sources: raw ingestion,
parsing, entity normalization, affinity reconstruction, review tooling, and
duplicate reporting — with every fact traceable to a source row and every
human decision durable across regeneration.

## Core rule

Never require information that the source does not contain. Missing fields
stay null; ambiguous records go to review, never to a guess.

## Repository layout

## Architecture: raw → parsed → normalized → affinity/report

1. **Ingest (CP3).** An external CSV becomes immutable `raw_source_rows`
   via the column mapping in `import_mappings.csv` — no per-format code.
   `source_record_id` is content-derived
   (`<source_id>:<sha256_16>:<occurrence>`), never positional. Each run's
   membership and order live in the append-only `import_runs`/`run_rows`
   ledger; the latest completed run defines a source's *current version*.
   Raw rows are write-once: corrections happen downstream, never in place.
2. **Parse (CP4).** The current version is classified row-by-row within
   subject blocks: `attribute` (registered labels only), `affinity_header`,
   `affinity_group` (approved delimiter only), `pairing_candidate`, or
   `unclassified` for review. Strength evidence resolves through
   `strength_mappings.csv` marker keys (`explicit_label:<value>`,
   `asterisk_uppercase`, `uppercase`, `plain`); a typography-lossy format's
   plain text gets a **null** score — no defaults, ever.
3. **Normalize (CP5).** Source strings resolve to entities (order below);
   derived `entity_attributes` and `pairing_observations` are rebuilt per
   source with full provenance. Unresolved names stay NULL.
4. **Affinities & reports (CP6).** `affinity_group` rows become
   `affinity_groups` + `affinity_members` (split only on the approved
   delimiter; members keep exact raw tokens and source order). Affinities
   are never flattened into binary pairings. Duplicate and reverse-pair
   reports are read-only: they flag, never merge or delete.
5. **Sample & CI (CP7).** The committed sample package is regenerated by
   the real pipeline, validated, and drift-checked in CI.

## Decision tables vs derived tables vs ledger

- **Configuration files** (human-authored inputs): `sources.csv`,
  `import_mappings.csv`, `strength_mappings.csv`, `attribute_labels.csv`,
  `affinity_split_rules.csv`.
- **Decision tables** (human-owned, durable):
  - `entities.csv` — **human-owned and never regenerated**. The pipeline
    imports it as an immutable reviewed input and verifies it is unaltered.
    New entities enter only through explicit reviewed decisions
    (`flavor_pairing.normalize.create_reviewed_entity` or a reviewed edit).
  - `entity_source_names.csv` — **durable mappings: protected human
    decisions plus deterministic machine mappings.** Machine rows
    (`unresolved`/`auto_mapped`) may be inserted or updated by reruns;
    rows with `human_mapped` or any unrecognized legacy status are
    human-owned and are never modified (fail-closed).
- **Generated tables** (pipeline output — **never hand-edit**):
  `raw_source_rows.csv`, `parsed_source_rows.csv`, `entity_attributes.csv`,
  `pairing_observations.csv`, `affinity_groups.csv`, `affinity_members.csv`.
  Together with `entity_source_names.csv` these are the seven files
  `scripts/regenerate_sample.py` exports.
- **Ledgers**: real ingestion runs append operational history under
  `data/ledger/` (versioned, public-safe sources) or the gitignored
  private path (restricted sources). Sample regeneration instead uses a
  temporary ledger in its scratch directory — a reproducible demo build,
  not operational history; the two serve different purposes.

## Entity-resolution order (exact)

For each `(source_id, source_text, source_role)` key — roles are
`subject`, `pairing_entry`, and `affinity_member`:

1. An existing `entity_source_names` row for the exact key (alias path).
   Human-owned rows are used as-is and never modified.
2. Exact canonical-name match: trim + case-fold only, no fuzzy/plural
   matching, entities with `review_status = rejected` excluded; exactly
   one candidate resolves, ambiguity does not.
3. Otherwise **unresolved**: `entity_id` NULL, status `unresolved`. No
   placeholder entities; no automatic entity creation of any kind.

## Review workflow

`flavor_pairing.review.build_review_queue` reports (read-only,
deterministic order): unresolved mappings (all NULL-entity rows, any
status), unclassified parsed rows, `requires_review` parsed rows, entities
needing review, observations with unresolved paired entities, and affinity
groups/members with unresolved subjects/members. A human resolves an item
by editing the relevant decision table (e.g. setting `entity_id` +
`normalization_status=human_mapped` on a mapping), then rerunning
normalization: the decision propagates into rebuilt derived rows and the
queue shrinks. Decisions survive full regeneration by construction.

## Duplicate and reverse-pair reports

`flavor_pairing.dupes`: exact duplicate observations (same resolved entity
pair, NULL never equals NULL), reverse-pair candidates (one row per
unordered pair with both directional observation sets intact — no score
merging, no symmetry inference, self-pairs excluded), and raw-text
duplicates (exact repeated `paired_text_raw`, including unresolved rows).
All read-only.

## Commands

```bash
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements-dev.txt

# full test suite
python3 -m pytest

# regenerate the canonical sample package (writes the seven generated
# files into data/sample; entities.csv and configuration are untouched)
python3 scripts/regenerate_sample.py

# verify the committed sample regenerates byte-identically (CI mode;
# modifies nothing, exits non-zero on drift)
python3 scripts/regenerate_sample.py --check

# validate the committed sample package
python3 scripts/validate_sample.py
```

Regeneration is deterministic: content-derived IDs, a fixed clock, and
sorted exports make independent runs byte-identical. `--input-dir`,
`--decision-dir`, `--output`, and `--scratch-dir` allow safe runs against
temporary copies. Public sample inputs are deliberately named
`data/sample_input/<source_id>.csv`. The one-time
`--bootstrap-legacy-mappings` flag existed to migrate the pre-CP7
duplicated `entity_source_names.csv`; ordinary regeneration is strict and
needs no flag.

## Continuous integration

`.github/workflows/tests.yml` runs on pull requests and on pushes to
`main` and `feature/data-foundation`: compileall → pytest →
`regenerate_sample.py --check` (fails if the committed sample drifts from
what the pipeline produces) → `validate_sample.py` → `git diff --check`.
Python 3.12, no secrets, no private data, no deployment.

## Private-data boundary

Restricted source **payloads** and anything under `data/imports_private/`
are never committed, never processed by sample regeneration, and never
fetched externally. (The source registry `sources.csv` and
`docs/DATA_SOURCES.md` may legitimately hold *metadata* about restricted
sources — provenance and rights status — without their data.) Enforcement
is layered: rights-aware ledger routing fails closed for any unrecognized
`rights_status`; sample regeneration rejects private paths and non-public
sources outright; validation rejects the private path marker in any
sample cell; and a meta-test forbids runtime references to the private
path.

## How to add a new flat tabular source

1. Register it in `sources.csv` (id, format, honest `rights_status`).
2. If the format is new: add its column mappings to `import_mappings.csv`,
   its marker rows to `strength_mappings.csv`, and (if applicable)
   `attribute_labels.csv` rows and an approved `affinity_split_rules.csv`
   rule. No code changes for flat tabular sources.
3. Ingest with `scripts/import_to_raw.py --source-id <id> input.csv out.csv`
   (restricted sources route their ledger to the gitignored private path).
4. Parse/normalize via the pipeline; work the review queue.
   For the public demo sample specifically: add
   `data/sample_input/<source_id>.csv` and rerun
   `scripts/regenerate_sample.py`.

## How to add or change a reviewed mapping

1. If the target entity is new, add it to `entities.csv` (or use
   `create_reviewed_entity`): caller-supplied `entity_id`, `entity_type`
   only as positively known (`unknown` otherwise), `normalization_status`
   `human_mapped`.
2. In `entity_source_names.csv`, set the row's `entity_id` and
   `normalization_status=human_mapped` (keep `source_name_id`,
   `source_text`, `source_role` unchanged).
3. Rerun normalization/regeneration: the machine respects the row
   verbatim; derived rows follow it; validation confirms.

## Out of scope for this phase (future app/API work)

Consumer-facing application and API, recommendation ranking, the derived
`pairing_edges` aggregation, fuzzy matching, automatic entity creation,
score aggregation, chemical/compound enrichment, PostgreSQL migration
(portability is preserved), and any processing of rights-unresolved
external datasets.

## Fields deliberately removed from the required core

Vegan/vegetarian status, allergens, numerical taste scores, ingredient
form/processing state, pairing explanations, per-pair context/technique,
confidence/evidence counts, chemical similarity — the referenced pairing
sources do not consistently provide them
(`docs/CORRECTION_FROM_EARLIER_PACKAGE.md`). They may return later through
separate, cited enrichment sources.

## Strength warning

Two-column GitHub-style extractions preserve uppercase and some asterisks,
but ordinary lowercase text does not prove whether the original was plain
or bold. `strength_score` therefore stays blank for such rows. Do not
convert lowercase rows to score 1.
